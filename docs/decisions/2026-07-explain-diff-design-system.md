# 2026-07 — explain-diff / micro-world 공유 디자인 시스템

## 상태

확정 (2026-07-18). 예정된 두 스킬(`explain-diff`, `micro-world`)이 생성하는 모든 HTML 산출물은
아래 단일 디자인 토큰 세트를 따른다. 기준 시안: **B v2.3** (Toss Card 계열, 사용자 확정).

## 배경

Geoffrey Litt(Notion)의 `/explain-diff` 개념을 도입하기로 했다: 코드 변경을 raw diff가 아닌
literate diff(산문 + 임베드 코드 + 인터랙티브 요소 + 이해도 퀴즈)로 설명하는 HTML 문서를
생성하는 스킬. micro-world(변경 전용 일회성 인터랙티브 시뮬)는 **별도 스킬**로 분리한다 —
원작자도 두 기법을 분리했고, explain-diff는 구조까지 정형, micro-world는 룩만 정형이기 때문.

두 스킬의 산출물이 일관된 룩앤필을 가지려면 공유 토큰이 필요했다. 시안 3종
(A: Editorial Ink / B: Toss Card / C: Graphite)을 제작·비교했고 사용자가 B를 선택,
이후 TDS(Toss Design System) 공개 문서의 패턴을 참고해 3차례 개정(v2.1→v2.3)했다.

> 넓은 surface가 필요하면 gray25 + gray100 보더를 쓴다.

### 이동·모션

- **`scroll-behavior: smooth`를 쓰지 않는다.** 문서가 길어 목차에서 하단 섹션으로 점프하면
  브라우저가 거의 고정 duration으로 애니메이션하므로 중간 구간을 훑고 지나가 어지럽다.
  이동은 즉시, 위치 인지는 착지 하이라이트(`.card:target` 1s 링)로 준다
- 섹션 착지 여백은 `scroll-margin-top: 24px`. 뷰포트 최상단에 붙으면 뚝 떨어진 느낌이 난다
- 하단 고정 바처럼 `position: fixed` + gradient + `color-mix` 조합은 스크롤 내내 재합성되므로
  `contain: paint`로 레이어를 분리한다
- `prefers-reduced-motion`에서는 transition·animation·transform을 모두 끈다

### 형태·타이포

- 라운드 3단: 카드 20px / 요소 14px / 버튼 10px
- 섀도: `0 1px 2px rgba(25,31,40,.03), 0 8px 24px rgba(25,31,40,.06)` (소프트 단일)
- 폰트: 시스템 스택 `-apple-system, "Apple SD Gothic Neo", "Pretendard", …` + 모노 `ui-monospace, "SF Mono", …`
  (Toss Product Sans는 전용 폰트라 사용 불가)
- 타이틀 700 / 섹션 700 / 버튼·본문 강조 600. 800은 쓰지 않는다 (과볼드 금지)
- letter-spacing `-0.01em ~ -0.03em`

### 컴포넌트 패턴 (TDS 패턴 재해석)

- **Top**: 필 eyebrow + 볼드 타이틀 + 리드문 + 메타 칩(pill)
- **ListHeader**: 넘버 스퀘어(22px, blue-soft) + 볼드 제목 + 트레일링 메타
- **List row**: 제목/설명 2줄 + 트레일링(chevron 또는 배지). 리딩 아이콘 없음
- **버튼**: fill(blue)/weak(white+blue) 2단 위계, 13px/600/패딩 8×14, `:active` scale(.97)
- **diff 블록**: gray25 배경, 추가=green-soft wash, 삭제=red-soft wash (저채도)
- **인터랙티브 피규어**: blue-soft 그라데이션 컨테이너 + 스텝 노드(활성 시 blue fill + lift)
- **퀴즈**: 선택지 리스트 행(gray25, hover 시 blue-soft), 정답 green/오답 red 판정 + 해설 공개
- **BottomCTA × 퀴즈 게이트**: 하단 고정 풀폭 CTA. 퀴즈 통과 전 gray200 비활성,
  전원 정답 시 blue 활성("이해 완료"). Litt의 speed regulator 규칙의 UI 구현

### 라이팅·금지 규칙

- **해요체 전면 적용**, 능동태, 긍정 프레이밍 (TDS UX 라이팅 원칙 차용)
- **이모지 금지** (파비콘 제외)
- **손그림 SVG 아이콘 금지** — 아웃라인·필드 두 차례 시도 후 "임의로 그린 티"로 기각.
  아이콘 없이 타이포·여백·배지로 위계를 만든다
- **외부 의존성 0** — self-contained HTML (인라인 CSS/JS, CDN·웹폰트·원격 이미지 금지)
- 산출물은 작업 저장소 루트에 저장하고 절대경로만 안내한다 (자동 열기 없음, 삭제는 사용자 지시 시)

## 산출물 구조 규칙 (2026-07-18 증보: 실 diff 샘플 v5로 검증)

실제 diff(chanmuzi-agent-harness PR #28)로 풀 샘플을 제작하고 사용자 피드백 5회를 거쳐
확정한 규칙. 토큰(위)이 "어떻게 보이는가"라면 이 절은 "어떻게 구성하는가"다.

### 표준 골격: 3파트

- **Part 1 개요** — 배경(왜) + 구조 한눈에 보기(주석 달린 디렉토리 트리 + 선택적 인터랙티브 피규어)
- **Part 2 변경 사항** — 테마별 카드: 산문(동작 의미 중심) + verbatim diff 발췌 + 핵심 정리 카드
- **Part 3 이해 점검** — 리스크(Watch/Critical) + 이해도 퀴즈
- 카드 묶음 사이에 파트 라벨(`Part N` 모노 + 그룹명 + 헤어라인), 목차에도 같은 그룹 헤더
- 목차 각 행 = 해당 섹션의 **한 줄 요약** (카테고리 라벨 금지), 앵커 클릭 이동

### 인터랙션 원칙

- **입력이 결과를 바꿀 때만** 인터랙션을 만든다. 이미 보이는 내용을 순서대로 켜는
  스텝 공개형("다음 단계" 버튼)은 이해에 기여하지 않으므로 금지
- 검증된 예: 시작 위치 토글 하나에 두 에이전트 열이 동시에 반응하는 로딩 시뮬레이터

### 퀴즈 규격

- 기본 5문항, 중~상 난이도. 트리비아 금지 — 행동 예측 / 실패 모드 / 기각된 대안의 근거 /
  불변식 / 미래 유지보수자 시나리오
- 오답지는 그럴듯한 흔한 오해로 구성 (옛 체계의 기억, 잘못된 레이어 귀속, 조건 바꿔치기 등)
- **오답 → 정답 비공개**, 관련 섹션을 가리키는 힌트 + "다시 풀기" (해당 문항만 리셋)
- 힌트 박스는 **중립 회색**(gray25 + gray100 보더). 오답 신호는 `.opt.wrong`(빨강)이 이미
  주므로 경고색을 겹쳐 쓰지 않는다. blue-soft는 해결된 상태(해설 카드) 전용으로 남긴다 —
  회색 = 진행 중, 파랑 = 해결됨의 위계
- 힌트는 **클릭 가능해야 한다**. `data-hint-href`(섹션 id) + `data-hint-label`(섹션 제목)로
  해당 섹션으로 보내는 solid 링크를 만든다. 손으로 스크롤해 되짚게 두지 않는다.
  id가 문서에 없으면 링크를 제거하고 텍스트 힌트만 남긴다 (깨진 링크 금지)
- 힌트 링크로 퀴즈를 떠난 경우에만 하단에 "퀴즈로 돌아가기" 칩을 띄운다. 단순 스크롤로
  벗어난 것은 이탈이 아니므로 띄우지 않는다
- **정답 → 해설 카드 + 잠금**. 해설은 라벨 + 핵심 한 줄(볼드) + 불릿 2~3개 (오답이 왜
  오답인지 포함)
- 하단 고정 게이트는 전원 정답 시에만 활성. **가짜 액션 문구 금지** — "PR 보내기"처럼
  실제 로직이 없는 표현 대신 "이 변경을 이해했어요" 같은 정직한 상태 표현만

### 컴포넌트 세부 (샘플에서 조정 확정)

- **핵심 정리 카드**: 섹션 마무리 요약은 산문이 아니라 퀴즈 해설과 동일한 시각 언어
  (blue-soft 카드 + 대문자 라벨 + 핵심 한 줄 + 불릿)로. 문서 전체에서 "파랑 틴트 카드 =
  기억할 것"이라는 규칙 성립
- **배지/칩 슬림 규격**: 웨이트 600, 10~10.5px, 패딩 1~2px 세로 — 본문/트리 행 높이를
  넘지 않는다 (700 웨이트 굵은 배지 금지)
- **세그먼트 컨트롤**: 연회색 트랙(잉크 5.5% 틴트) 위 버튼. 비활성은 gray600 텍스트 +
  hover 시 흰 배경·파랑 텍스트로 반응(잠긴 인상 방지), 활성은 blue fill + 파랑 섀도
- **디렉토리 트리**: 카드 상단 헤더(타이틀 좌 + legend 칩 우) + 본문 행(경로 + 슬림 칩).
  삭제 파일은 취소선. 섹션 점프 링크는 달지 않는다 (배지만으로 충분)

### 배포·산출물 규칙

- 산출물은 작업 저장소 루트에 `explain-diff-<slug>.html`로 저장하고 **절대경로만 안내**
  (자동 열기 없음, 커밋하지 않음, 삭제는 사용자 지시 시)
- 스킬 폴더에 `template.html`(토큰·컴포넌트·퀴즈 JS 포함)을 동봉해 배포한다 —
  Skills CLI는 스킬 폴더만 복사하므로 저장소의 별도 문서는 설치본에 실려가지 않는다

### micro-world 커맨드 센터 문법 (2026-07-18 증보 2: 샘플 v7로 검증)

micro-world 산출물의 프레임 규칙. 실 소재(플러그인 cache 라이프사이클)로 7개 버전을 돌며 확정했다.

- **가이드 단일 사이클**: 저작된 시나리오 하나를 미션 바(목표 체크 칩 2~3개)로 프레이밍하고,
  매 순간 권장 액션 하나만 활성화·하이라이트한다. 자유 탐험 모드는 의도적으로 없다 —
  비가이드 분기는 이해를 희석한다. 시나리오는 가르치려는 실패(함정)를 통과해 복구하는 경로로 짠다
- **인라인 재시작**: 재시작은 액션 팔레트 안에 같은 문법(`↻`)으로 두고, 미션 완료 시
  권장 하이라이트를 받는다. 컨트롤에서 떨어진 리셋 링크 금지
- **재생 컨트롤**: 액션 하나 = 스냅샷 하나. 환경 헤더의 `‹ ›`로 이동하고,
  **앞 히스토리가 없으면 `›`가 권장 다음 액션을 직접 실행**한다 — 첫 화면에서 `→` 연타만으로
  전체 시나리오를 완주할 수 있어야 한다. 키보드 `←`/`→`/`R`(한글 자판 `ㄱ` 포함) 지원
- **사실/해설 분리**: 환경(터미널 등)에는 실제 시스템이 출력할 내용만 담는다 — 판정 색·해설 금지.
  의미는 환경 바로 아래 해설 스트립("지금 무슨 일이": 핵심 한 줄 볼드 + 불릿 최대 2개,
  스트립 전체 시맨틱 틴트)에만 둔다
- **세계 상태 통합 카드**: 관찰 가능한 상태는 "세계 상태" 카드 하나에 내부 구획으로 모은다
  (흩어진 형제 카드 금지). 값 변경 시 펄스/등장 애니메이션으로 동작↔상태 연결을 보여준다
- **embodiment는 소재가 결정**: CLI 절차 → 터미널 커맨드 센터 / 실행 추적 → 타임라인 스크럽 /
  구조 개편 → 드래그 앤 드랍 + 양측 구조 병렬 시각화 / 수식·임계값 → 파라미터 노브.
  이미 보이는 내용을 순서대로 켜는 스텝 공개형은 여전히 금지
- **충실도**: 실제 로직의 구분(예: "복사는 디스크만, reload가 세션을 바꾼다")이 모델에
  존재해야 한다. 단순화는 경고 콜아웃이 아니라 **하단 회색 각주**로 고지
- **스코프**: 혼란 하나 = 월드 하나. 혼란이 여럿이면 페이지를 키우지 말고 월드를 쪼갠다
- 기준 샘플: https://claude.ai/code/artifact/59197952-3ba9-4c58-a1f0-a0b64becbcc8 (버전 라벨 sample-command-center-v7-play)

## 검토한 대안

- **A: Editorial Ink** (애플 문서 계열, 카드 없는 타이포 중심) — 사용자 선호로 기각
- **C: Graphite** (모던 다크) — 동일
- **아웃라인 SVG 아이콘 → 필드 SVG 아이콘 → 아이콘 제거**: 실제 TDS 아이콘은 라이선스상
  사용 불가, 손그림 근사치는 품질 미달. "못생긴 아이콘보다 아이콘 없음"으로 확정

## 법적 주의

TDS 자산(UI Kit, 아이콘, Toss Product Sans)은 토스 지식재산으로 앱인토스 파트너 라이선스
대상이다. 이 디자인 시스템은 **공개 문서의 원칙·패턴을 참고한 독자 구현**이며 TDS 자산을
포함하지 않는다. 토큰 값은 자체 정의다.
